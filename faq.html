FAQ

Mu-referencing

Do I need to write explicit MU-referencing like in NONMEM or nlmixr2?

No. You do not need to write an explicit MU_i intermediate variable. ferx detects the same structure automatically from the right-hand side of each [individual_parameters] line.

In NONMEM and nlmixr2, MU-referencing is required in SAEM because the conjugate-Gibbs E-step is only valid when the ETA→THETA relationship is strictly linear on a transformed scale. In ferx, you simply write the individual-parameter expression naturally:

CL = TVCL * exp(ETA_CL)              # detected: multiplicative / log-normal
CL = exp(log(TVCL) + ETA_CL)         # detected: canonical MU form
CL = TVCL * (WT/70)^0.75 * exp(ETA_CL)  # detected: with covariate
CL = TVCL + ETA_CL                   # detected: additive

The parser records the ETA→THETA mapping and re-centres the inner-loop ETA search at each outer iteration.

Patterns not auto-detected (falls back to eta = 0 start):

  • Two or more free thetas in the product (TVCL * TVSCALE * exp(ETA_CL)) — ambiguous anchor
  • Compound eta expressions inside exp (exp(ETA_CL + ETA_OCC))

Mu-referencing affects convergence speed and robustness, not the final MLE. Disable with mu_referencing = false in [fit_options] to reproduce pre-automatic behaviour.

Conditional logic

Can I use if / else statements like in NONMEM or nlmixr2?

Yes. Two forms are supported in [individual_parameters] and [odes]:

# Block form
if (WT > 70) {
  CL = TVCL * (WT / 70)^0.75 * exp(ETA_CL)
} else if (SEX == 1) {
  CL = TVCL * 1.2 * exp(ETA_CL)
} else {
  CL = TVCL * exp(ETA_CL)
}

# Inline (ternary) form
CL = if (SEX == 1) TVCL * 1.5 else TVCL

Compared to NONMEM: C-style {braces} and operators (>, ==, &&) rather than Fortran-style IF THEN … ENDIF and .GT., .LE., etc.

Compared to nlmixr2: near-verbatim match — replace <- with =.

Caveat: == and != do bitwise float comparison. They work for integer-coded covariates (SEX == 1) but are unreliable for continuous thresholds. Prefer WT >= 70 && WT < 80 over WT == 75.

Caveat: when a parameter is assigned inside an if block, mu-reference detection is skipped for that parameter. See Individual Parameters.

Optimizer selection

Which outer optimizer should I pick?

slsqp (the default) is the right choice for most models — it is fast, handles box constraints cleanly, and behaves well on the log-transformed parameter scale.

Reach for a different optimizer when SLSQP misbehaves:

  • bobyqa — derivative-free, robust when FOCE’s FD gradients are noisy and SLSQP stalls or oscillates. Slower per iteration on smooth problems.
  • trust_region — second-order Newton trust-region. Can be faster near convergence; tune the CG budget with steihaug_max_iters (default 50) for models with more than ~50 packed parameters.
  • lbfgs / bfgs — fall back to these only when NLopt is unavailable.

See Fit Options — Optimizer choices.

Block omega

How do I switch from diagonal to block omega?

Each omega variance becomes a diagonal entry; off-diagonals start at 0.0:

# Before
omega ETA_CL ~ 0.1
omega ETA_V1 ~ 0.1
omega ETA_Q  ~ 0.1

# After (3×3 block, 6 values: 1+2+3)
block_omega (ETA_CL, ETA_V1, ETA_Q) = [
  0.1,
  0.0, 0.1,
  0.0, 0.0, 0.1
]

Block omega regularises poorly-identified parameters by letting them share variability. Starting all off-diagonals at 0.0 lets the fit recover the correlations from the data. See Parameters — Block omega.

Sigma scale

Is sigma estimated on the SD scale or the variance scale?

SD scale. sigma PROP_ERR ~ 0.1 for a proportional model is a 10% CV initial value.

The fit YAML emits both estimate (SD) and variance (SD²). For proportional components it also emits cv_pct = estimate * 100. See Error Model — Sigma scale.

FOCEI vs FOCE

When should I use FOCEI instead of FOCE?

Use FOCEI (default) whenever the error model is proportional or combined. FOCE evaluates the residual variance at η = 0 (population mean); FOCEI evaluates it at the individual EBE η̂, capturing the dependence between residual error and individual predictions.

For a purely additive error model, FOCE and FOCEI are equivalent (the interaction term is zero). Using FOCE for additive-error models can save a small amount of computation.

ODE vs analytical

When must I use an ODE model?

Use an ODE model when the kinetics cannot be expressed as a sum of exponentials:

  • Saturable (Michaelis-Menten) elimination
  • Target-mediated drug disposition (TMDD)
  • Transit compartment absorption
  • Custom multi-compartment structures

For standard first-order kinetics use the analytical models (one_cpt_oral, two_cpt_oral, etc.) — they are 10–100× faster and support Enzyme autodiff gradients. See ODE Models.

Porting from NONMEM

How do I port a NONMEM model to ferx?

  1. Drop all $ABBR, $MU_i, $SIZE boilerplate — not needed in ferx.
  2. Translate $PK assignments to [individual_parameters] expressions with = instead of =.
  3. Replace THETA(i), ETA(i), EPS(i) with descriptive names (TVCL, ETA_CL, PROP_ERR).
  4. If the NONMEM model uses explicit MU_i = LOG(THETA(i)) lines, drop them — ferx detects the mu-reference automatically.
  5. Map $SUBROUTINE ADVAN to the equivalent ferx pk function:
    • ADVAN1 → one_cpt_iv_bolus
    • ADVAN2 → one_cpt_oral
    • ADVAN3 → two_cpt_iv_bolus
    • ADVAN4 → two_cpt_oral
    • ADVAN11 → three_cpt_iv_bolus
    • ADVAN12 → three_cpt_oral
  6. Map $ERROR to [error_model]. Remember sigma is on the SD scale in ferx (not variance).

Gradient methods

When does Enzyme autodiff kick in, and when does ferx fall back to finite differences?

The gradient path is resolved per-subject at runtime:

Path When used
AdSingleSnapshot (Enzyme reverse-mode) Subject has no time-varying covariates; analytical PK model
AdEventDriven (Enzyme reverse-mode, event-driven) Subject has TV covariates; 1- or 2-cpt analytical model
Fd (central finite differences) FERX_NO_AUTODIFF=1 build, ODE model, or TV-cov on 3-cpt model

fit$gradient_used (in R) shows which path was resolved. See Autodiff for details.
